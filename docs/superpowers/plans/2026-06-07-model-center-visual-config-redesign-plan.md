# 模型中心可视化配置重设计实施计划

> **For agentic workers:** Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把“模型中心”改造成普通中文用户能看懂、能新增、能编辑、能应用、能删除配置的可视化配置管理器，默认不让用户直接面对 JSON 文件。

**Architecture:** 保留现有 `VisualModelConfigStore`、`VisualModelConfig`、`VisualStageConfig` 和底层 JSON 存储，把 UI 重组为“配置库 + 配置详情 + 阶段编辑抽屉 + 校验中心”。内置方案只读，自定义方案可编辑，所有表单操作最终仍映射到底层配置文件，避免重写 pipeline 和 adapter。

**Tech Stack:** Python 3.10、PySide6、Pydantic、现有 `ivo.core.visual_model_config`、`ivo.core.model_presets`、`ivo.ui.model_center`、`ivo.ui.advanced_model_settings`、`pytest-qt`。

---

## 1. 背景与范围

当前“模型中心”的核心价值已经明确：它不是一个普通设置页，而是对模型运行方案做增删改查，并把选中的方案应用到后续生成流程。底层配置目前以 JSON 保存，用户实际需要的是“我有哪些方案、哪个方案能用、每个阶段用什么模型、缺什么、能不能一键应用”。

本计划只聚焦模型中心页面，不重写配音流水线、不改模型下载策略、不改变项目库和生成页。目标是在现有能力基础上，把配置体验从“开发者表单”升级为“普通用户能操作的配置中心”。

## 2. 参考方向

本设计借鉴这些成熟模式，但不复制任何品牌视觉资产：

- Apple Human Interface Guidelines：强调层级、协调、一致性；设置项过多会降低亲和力，因此普通用户视图要做渐进披露。参考：[Human Interface Guidelines](https://developer.apple.com/design/human-interface-guidelines)、[Settings](https://developer.apple.com/design/human-interface-guidelines/settings)、[Split views](https://developer.apple.com/design/human-interface-guidelines/split-views)。
- Windows / Fluent 控件模式：配置类桌面工具适合使用列表、详情、表单、信息条、进度与状态徽标等标准控件。参考：[Controls for Windows apps](https://learn.microsoft.com/en-us/windows/apps/develop/ui/controls/)。
- Notion / Raycast / Windows 设置类产品的共同模式：左侧列表管理对象，右侧编辑详情；主操作固定在清晰位置；高级配置折叠，不污染普通流程。

## 3. 当前问题诊断

现有 `src/ivo/ui/model_center.py` 已经实现推荐方案按钮、自定义配置列表、阶段表格、模型目录检查和开发者设置折叠，但信息层级仍然混在一个纵向页面中：

- 推荐方案卡片、`我的配置` 列表、表单、阶段表格、模型目录和开发者设置全部顺序堆叠，用户无法判断先看哪里、先点哪里。
- 推荐方案和“我的配置”本质都是配置对象，但 UI 上被分成两个区域，用户会疑惑两者关系。
- `本地流程配置`、`翻译服务配置` 仍然暴露底层路径概念，普通用户不知道该填什么。
- 阶段表格只显示启用、阶段、运行方式、服务名称，缺少“模型是否找到、API 是否连通、当前参数摘要、为什么不可用”等决策信息。
- 新增、复制、保存、删除、应用按钮分散在页面中，没有形成清晰的 CRUD 主流程。
- 开发者设置虽然默认隐藏，但展开后仍是完整旧表单，缺少“普通配置”和“源文件调试”的边界说明。
- 校验结果没有聚合为当前配置的可用性状态，用户需要自己理解多个检查按钮的结果。

## 4. 设计原则

- **配置即对象**：内置方案和自定义方案都在一个配置库中展示，只用状态区分“推荐 / 我的 / 已应用 / 不可编辑”。
- **默认不见 JSON**：普通模式不出现 JSON、profile、adapter、命令、响应映射、KEY=VALUE 等技术词；这些只放进“高级源文件”抽屉。
- **先状态，后参数**：每个配置先告诉用户“能不能用、缺什么、适合什么”，再允许编辑细节。
- **阶段可视化**：用流程卡片替代表格作为主视觉，让用户知道人声分离、语音识别、说话人识别、翻译、语音合成分别怎么跑。
- **固定主操作**：页面始终只有一个最突出的主按钮：“应用此配置”或“保存并应用”。
- **内置只读，自定义可改**：内置推荐方案不能直接编辑，用户点击编辑时自动引导“复制为我的配置”。
- **高级能力保留**：高级用户仍能导入、导出、查看源文件路径和打开旧配置表单，但入口要明确标注“开发者调试”。

## 5. 推荐信息架构

模型中心页面改成三层结构：

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部状态栏：模型目录 · 当前应用配置 · 一键检查 · 搜索配置      │
├──────────────┬───────────────────────────┬────────────────┤
│ 配置库        │ 当前配置详情               │ 编辑抽屉/校验中心 │
│ 推荐          │ 名称、说明、状态            │ 基础信息          │
│ 我的配置      │ 阶段流程卡片                │ 阶段详情          │
│ 搜索/筛选     │ 需要的模型与服务            │ 模型/API 选择     │
│ 新建配置      │ 应用、复制、删除            │ 高级源文件        │
└──────────────┴───────────────────────────┴────────────────┘
```

窄窗口下改为两栏：左侧配置库可折叠，右侧详情和编辑抽屉上下排列。窗口高度变化时，底部主按钮固定在详情区底部，避免用户找不到保存或应用入口。

## 6. 核心页面区域

### 6.1 顶部状态栏

顶部只展示全局状态：

- 模型目录：显示当前目录，例如 `D:\临时软件包\IntelligentVoiceOver\models`，按钮为“更换目录”。
- 当前应用配置：显示配置名和状态，例如“本机高质量 · 已应用”。
- 一键检查：检查当前选中配置需要的模型、LM Studio、在线 API 和本地运行环境。
- 搜索配置：按配置名、阶段服务、模型名称搜索。

检查结果用状态胶囊展示：

- 绿色：可用。
- 橙色：缺少可选项或建议优化。
- 红色：不可运行。
- 灰色：尚未检查。

### 6.2 配置库

配置库是所有 CRUD 的入口，分组展示：

- 推荐方案：内置只读，例如“本机高质量（LM Studio + F5-TTS）”“本机快速预览（GPU）”“CPU 快速预览”“自定义在线 API”。
- 我的配置：用户复制或新建的配置。
- 最近使用：最多显示 3 个最近应用过的配置。

每个列表项显示：

- 配置名称。
- 简短说明，一行省略。
- 状态点：可用、需检查、不可用、草稿、已应用。
- 标签：GPU、LM Studio、在线 API、本地模型、高质量、快速预览。

配置库顶部提供“新建配置”按钮，点击后出现菜单：

- 从推荐方案复制。
- 从空白开始。
- 从本地源文件导入。

### 6.3 当前配置详情

详情区展示选中配置的完整摘要：

- 配置名、说明、适用场景。
- 是否内置、是否已应用、是否有未保存修改。
- 阶段流程卡片。
- 推荐模型与服务。
- 最近检查时间。
- 主要操作：应用此配置、复制为我的配置、编辑、删除。

内置配置的详情页不直接显示可编辑输入框。点击“编辑”时弹出确认：“推荐方案是只读的，已为你复制一份可编辑配置。”

### 6.4 阶段流程卡片

用横向或纵向流程卡片替代主表格：

1. 人声分离
2. 语音识别
3. 说话人识别
4. 翻译
5. 语音合成

每张卡片显示：

- 阶段开关。
- 运行方式：本地模型、在线 API、跳过。
- 当前模型或服务名称。
- 状态：已就绪、未检查、缺模型、API 未连通、配置不完整。
- 参数摘要，例如“CUDA · float16 · faster-whisper large-v3”。
- 操作：“编辑阶段”“测试此阶段”。

说话人识别允许标记为“可选但推荐”。如果关闭，卡片要说明影响：“关闭后会降低多角色声音一致性。”

### 6.5 编辑抽屉

编辑抽屉从右侧打开，不把所有字段一次堆在主页面。抽屉分为三个 Tab：

- 基础信息：名称、说明、适用内容类型、质量偏好、是否优先 GPU。
- 阶段设置：逐阶段选择本地模型、在线 API 或跳过。
- 高级源文件：只给开发者查看和修改底层源文件路径。

抽屉底部固定操作：

- 保存。
- 保存并应用。
- 放弃修改。

有未保存修改时，切换配置、关闭抽屉、离开页面都要弹出确认。

## 7. 可视化编辑字段设计

### 7.1 基础信息

字段：

- 配置名称：必填，示例“我的高质量日剧配置”。
- 配置说明：可选，示例“翻译走 LM Studio，其余阶段使用本地 GPU 模型”。
- 质量偏好：快速预览、高质量、质量优先。
- 硬件偏好：优先使用 GPU、允许 CPU 兜底。
- 内容类型：美剧、日剧、韩剧、通用。

### 7.2 本地模型阶段

当阶段选择“本地模型”时，字段为：

- 模型类型：根据阶段限定选项，例如语音识别只显示 faster-whisper 相关选项。
- 模型位置：默认跟随全局模型目录，也可以为单阶段指定路径。
- 推理设备：自动、CUDA、CPU。
- 精度：自动、float16、int8。
- 本地运行环境：默认使用包内环境；高级用户可改。

用户不需要选择命令文件。UI 根据选择映射到已有本地流程配置，或在后续阶段生成对应源配置。

### 7.3 在线 API 阶段

当阶段选择“在线 API”时，字段为：

- 服务名称：例如“LM Studio Qwen 翻译”。
- API 地址：示例 `http://127.0.0.1:1995/v1`。
- 模型名称：支持从接口读取模型列表；读取失败时允许手动填写。
- API Key：可选，默认隐藏内容。
- 测试连接：只测试当前阶段，不启动完整生成。

高级映射字段折叠在“响应格式设置”里，默认使用内置模板。只有用户选择“自定义响应格式”时才显示字段映射。

### 7.4 高级源文件

高级源文件区显示但默认折叠：

- 本地流程源文件。
- 翻译服务源文件。
- 人声分离 / 语音识别 / 说话人识别 / 语音合成在线服务源文件。
- 导入源文件。
- 导出当前配置。
- 打开源文件所在目录。

这一栏可以出现 JSON、profile、adapter 等技术词，因为它已经被标记为开发者区域。

## 8. CRUD 交互流程

### 8.1 新增配置

1. 用户点击“新建配置”。
2. 选择“从推荐方案复制”或“从空白开始”。
3. UI 创建草稿配置，名称默认为“新的模型配置”。
4. 自动打开编辑抽屉。
5. 用户填写基础信息并设置阶段。
6. 点击“保存”后写入 `model-configs.json`。
7. 点击“保存并应用”后同时发出 `preset_applied`。

### 8.2 查看配置

1. 用户在配置库中点击配置。
2. 详情区显示配置摘要。
3. 如果该配置有未检查项，顶部提示“建议先检查此配置”。
4. 如果配置不可用，详情区列出具体原因和修复按钮。

### 8.3 编辑配置

1. 用户点击“编辑”。
2. 如果是自定义配置，打开编辑抽屉。
3. 如果是内置配置，提示复制为自定义配置。
4. 保存后更新配置库列表、详情摘要和状态。
5. 保存失败时显示错误原因，不关闭抽屉。

### 8.4 删除配置

1. 只有自定义配置显示“删除”。
2. 点击后弹出确认：“删除后不会影响已生成项目，只会移除这份模型配置。”
3. 删除后选中默认推荐方案。
4. 如果删除的是当前已应用配置，自动回退为默认推荐方案，并提示用户重新应用。

### 8.5 应用配置

1. 用户点击“应用此配置”。
2. 如果有未保存修改，按钮文案变为“保存并应用”。
3. 如果未检查，提示“建议检查后应用”，但允许用户继续。
4. 应用成功后更新全局设置和主窗口当前配置。
5. 生成项目时读取当前应用配置，不要求用户再选择源文件路径。

## 9. 数据映射设计

现有模型继续保留：

- `VisualModelConfigStore`：负责读取内置配置与自定义配置。
- `VisualModelConfig`：负责配置级信息。
- `VisualStageConfig`：负责阶段级信息。

需要扩展字段：

- `VisualModelConfig.quality_label: str`
- `VisualModelConfig.prefer_gpu: bool`
- `VisualModelConfig.content_types: list[str]`
- `VisualModelConfig.last_check_status: str`
- `VisualModelConfig.last_check_summary: str`
- `VisualStageConfig.model_path: str`
- `VisualStageConfig.device: str`
- `VisualStageConfig.precision: str`
- `VisualStageConfig.api_base_url: str`
- `VisualStageConfig.api_model: str`
- `VisualStageConfig.validation_status: str`
- `VisualStageConfig.validation_message: str`

第一阶段可以只新增 UI 所需字段并保持向后兼容。旧配置加载时缺省字段使用默认值，不破坏已有 `model-configs.json`。

底层源文件映射策略：

- 内置推荐方案继续由 `src/ivo/core/model_presets.py` 指向已有示例源文件。
- 自定义配置保存到 `model-configs.json`。
- 普通字段先作为可视化 metadata 保存。
- 真正运行时仍使用已有 `local_profiles_path` 和各阶段 HTTP profile path。
- 后续再增加“根据可视化字段生成本地流程源文件”的能力，这不是本次 UI 重设计的第一优先级。

## 10. 视觉规范

整体风格：克制、明亮、干净，接近现代 macOS 设置类软件，但保持 Windows 桌面可用性。

- 背景：`#F5F5F7`。
- 页面面板：`#FFFFFF`。
- 主色：`#007AFF`。
- 成功：`#34C759`。
- 警告：`#FF9500`。
- 错误：`#FF3B30`。
- 文字主色：`#111827`。
- 次级文字：`#6B7280`。
- 分割线：`#E5E7EB`。
- 卡片圆角：10-12px。
- 输入框圆角：8px。
- 页面外边距：24-28px。
- 卡片内边距：14-18px。

控件层级：

- 主按钮：只用于“应用此配置”或“保存并应用”。
- 次按钮：编辑、复制、检查、测试连接。
- 危险按钮：删除配置，放在详情页底部或更多菜单，不与主按钮并列抢焦点。

列表样式：

- 配置库使用紧凑列表，不再用大块重复卡片堆叠。
- 选中项使用浅蓝背景和左侧强调条。
- 状态使用小胶囊，不使用长篇说明。

## 11. 文件结构计划

### 修改文件

- `src/ivo/ui/model_center.py`：重构为页面容器，组合配置库、详情区、编辑抽屉和校验中心。
- `src/ivo/core/visual_model_config.py`：扩展可视化配置字段，保持旧配置兼容。
- `src/ivo/core/model_presets.py`：为推荐方案补充质量、标签、默认阶段摘要和说明。
- `src/ivo/ui/advanced_model_settings.py`：改为开发者源文件面板，减少普通页面暴露。
- `src/ivo/ui/theme.py`：补充模型中心列表、状态胶囊、抽屉、阶段卡片样式。
- `src/ivo/ui/empty_states.py`：增加配置为空、检查未运行、配置不可用的空状态。
- `src/ivo/ui/main_window.py`：保证应用配置后主窗口状态同步。

### 新增文件

- `src/ivo/ui/model_center_components.py`：拆分配置库、详情卡、阶段卡、编辑抽屉等小组件。
- `tests/ui/test_model_center_visual_crud.py`：覆盖新建、复制、编辑、保存、删除、应用。
- `tests/ui/test_model_center_stage_editor.py`：覆盖阶段卡片、阶段抽屉、运行方式切换。
- `tests/ui/test_model_center_validation_panel.py`：覆盖校验状态展示和错误提示。
- `tests/core/test_visual_model_config_migration.py`：覆盖旧配置兼容加载。

## 12. 分阶段实施计划

### Phase 1: 数据模型与兼容测试

**目标：** 先让可视化字段有稳定数据来源，同时保证旧配置能读取。

- [ ] 给 `VisualModelConfig` 和 `VisualStageConfig` 增加可选字段和默认值。
- [ ] 编写旧 `model-configs.json` 加载测试，确认缺少新字段不会报错。
- [ ] 为内置配置补充标签、质量偏好、推荐模型说明。
- [ ] 运行 `uv run pytest tests\core\test_visual_model_config.py tests\core\test_visual_model_config_migration.py -q`。

### Phase 2: 配置库与详情页重构

**目标：** 把推荐方案和我的配置合并为统一配置库，建立清晰查看与应用流程。

- [ ] 拆出 `ConfigLibraryPane`，展示推荐、我的配置、最近使用。
- [ ] 拆出 `ConfigDetailPane`，展示配置摘要、状态、推荐模型和主操作。
- [ ] 内置配置详情默认只读。
- [ ] 自定义配置显示编辑和删除入口。
- [ ] 应用配置后更新状态胶囊和 `preset_applied` 信号。
- [ ] 运行 `uv run pytest tests\ui\test_model_center.py tests\ui\test_model_center_visual_crud.py -q`。

### Phase 3: 阶段流程卡片与编辑抽屉

**目标：** 用可理解的阶段卡片替代表格主视觉，让用户逐阶段选择本地模型、在线 API 或跳过。

- [ ] 新增 `StageFlowEditor`，按阶段顺序渲染流程卡片。
- [ ] 新增 `StageEditorDrawer`，支持基础信息、阶段设置和高级源文件三个 Tab。
- [ ] 阶段运行方式切换时动态显示本地模型字段或在线 API 字段。
- [ ] 内置配置点击编辑时自动复制为自定义配置。
- [ ] 有未保存修改时显示“草稿”状态。
- [ ] 运行 `uv run pytest tests\ui\test_model_center_stage_editor.py -q`。

### Phase 4: 校验中心与错误解释

**目标：** 把“刷新模型目录检查”“检查本地模型是否就绪”“校验在线 API 配置”聚合成一个用户能读懂的检查结果。

## 13. 验收标准

- 普通模式下，用户可以在 3 步内应用一个推荐配置：选择配置、检查、应用。
- 普通模式下不出现 JSON、profile、adapter、命令、KEY=VALUE、响应映射等技术词。
- 用户可以新增、复制、编辑、保存、删除、应用自定义配置。
- 内置配置不可直接修改，编辑时引导复制。
- 每个阶段都能看出启用状态、运行方式、模型或服务、可用性。
- 校验结果按阶段聚合，错误提示能告诉用户下一步怎么做。
- 当前已应用配置在配置库和顶部状态栏都有清晰标识。
- 展开高级源文件后，开发者仍能导入、导出和校验底层源文件。
- 窗口缩放时页面不会卡死、按钮不会消失、布局不会重叠。
- 所有新增核心交互都有 pytest-qt 覆盖。

## 14. 风险与控制

- **风险：字段扩展破坏旧配置。** 控制：所有新字段设默认值，并增加旧配置迁移测试。
- **风险：视觉重构影响现有按钮测试。** 控制：先保留现有公共方法，例如 `current_config_id()`、`apply_current_preset()`、`advanced_settings_visible()`，再逐步替换内部 UI。
- **风险：普通模式隐藏太多，高级用户难以调试。** 控制：高级源文件区保留完整旧能力，并提供“打开源文件目录”。
- **风险：校验逻辑跨本地模型、LM Studio 和在线 API，结果复杂。** 控制：第一版只做聚合展示，不改底层校验命令；后续再逐阶段增强。
- **风险：三栏布局在小窗口拥挤。** 控制：小于 1200px 时改为两栏，编辑抽屉变成底部面板。

## 15. 本计划不包含

- 不重写配音生成 pipeline。
- 不重新设计项目库和生成进度页。
- 不改变模型下载目录策略。
- 不主动压缩便携包。
- 不把桌面端改成 Web 或 Electron。
- 不在普通模式中提供手写 JSON 编辑器。

- 已覆盖用户提出的“模型中心本质是 JSON 配置增删改查”的需求。

- 已保留现有底层 JSON 存储和运行配置，避免大范围重写。
- 已规定普通用户不直接面对 JSON，高级用户仍能访问源文件。
- 已给出文件结构、阶段任务、测试命令和验收标准。
